*neovim-codex.txt*  NeoVim Codex integration

==============================================================================
INTRODUCTION                                                  *neovim-codex*

neovim-codex is a NeoVim-hosted client for `codex app-server`.

==============================================================================
COMMANDS                                               *neovim-codex-commands*

:CodexStart
    Start `codex app-server` and complete the initialize handshake.

:CodexStop
    Stop the running app-server process.

:CodexStatus
    Print the current connection state and active thread id.

:CodexEvents
    Open the protocol event log in the stacked viewer layer.

:CodexSmoke
    Run the current smoke checks and open a report buffer.

:CodexChat
    Open or focus the side rail.

:CodexChatRail
    Open the narrow side-rail shell explicitly.

:CodexChatOverlay
    Open the centered overlay explicitly.

:CodexChatReader
    Backward-compatible alias for :CodexChatOverlay.

:CodexSend
    Send the current composer contents, or open compose review when staged
    fragments exist.

:CodexThreadNew
    Start a fresh thread and make it active.

:CodexThreadNewConfig
    Start a fresh thread through the runtime settings flow.

:CodexThreads
    Pick and resume a stored thread.

:CodexThreadRead [thread-id]
    Read a thread into a report buffer without resuming it.

:CodexThreadRename [name]
    Rename the active thread, or prompt for a new name.

:CodexThreadFork [thread-id]
    Fork from a chosen turn in the active thread, or the supplied thread id.

:CodexThreadArchive [thread-id]
    Archive the active thread, or pick one to archive.

:CodexThreadUnarchive [thread-id]
    Restore an archived thread, or pick one from archived threads.

:CodexThreadSettings [thread-id]
    Edit sticky model, effort, approval policy, and collaboration mode for a
    thread.

:CodexThreadCompact [thread-id]
    Start manual history compaction for a thread.

:CodexInspect
    Open a details viewer for the selected transcript block.

:CodexInterrupt
    Interrupt the current running turn, if one exists.

:CodexSteer [text]
    Steer the current running turn, or use the active draft when the shell is
    open.

:CodexRequest
    Reopen the active pending approval or question request.

:CodexReview [request-key]
    Open the current pending file-change review, or reopen a pending
    file-change request by key.

:CodexShortcuts
    Open the Codex shortcut sheet for the current surface.

:CodexWorkbench
    Toggle the workbench tray for the active thread.

:CodexCompose
    Open compose review for the active thread.

:CodexCapturePath
    Stage the current file as a path fragment.

:CodexCaptureSelection
    Stage the current visual selection as a code fragment.

:CodexCaptureDiagnostic
    Stage the current diagnostic under cursor as a diagnostic fragment.

CHAT                                                           *neovim-codex-chat*

The current chat UI has two explicit shells:

- the default shell is a narrow right-side rail
- the overlay is a wider centered view of the same thread state
- both modes keep the markdown transcript above the multiline markdown
  composer

Run >
    :CodexChat
<
to open or focus the side rail.

Use >
    :CodexChatRail
    :CodexChatOverlay
    :CodexChatReader
<
when you want an explicit mode.

Usage:

1. Write in the composer.
2. Press <C-s> or run >
    :CodexSend
<
   If the workbench is non-empty, compose review opens first so you can place
   fragment handles such as [[f1]] into the packet template.
3. Use <CR> for newlines.
4. Use `q` inside the current shell to close it.

The transcript is protocol-first. It projects app-server item types into stable
surfaces instead of dumping shell output directly.

Examples:

- user messages and final responses stay primary
- commentary becomes markdown headings plus quoted working notes
- successful read/list/search commands become compact activity summaries
- failed or unknown commands stay terse in the transcript and open into a details popup
- file changes, tool calls, review mode, and context compaction each get their
  own summary surfaces

Default transcript mappings:

- `q` close the current chat shell
- `gr` reopen the active thread inbox
- `gR` switch between the side rail and the centered overlay
- `i` focus the composer
- insert-like keys (`a`, `A`, `i`, `I`, `o`, `O`, `R`) also jump to the composer instead of entering insert mode
- `<C-w>w` switch between transcript and composer without leaving the current shell
- `<CR>` inspect the selected transcript block in the stacked viewer layer
- `[[` jump to the previous turn
- `]]` jump to the next turn
- `gs` open the active thread settings sheet
- `g?` or `<F1>` open the shortcut sheet for the current surface

Default composer mappings:

- `<C-s>` send the current draft from normal or insert mode
- `gS` send the current draft from normal mode
- `gT` steer the running turn with the current draft
- `<C-w>w` in normal mode switches back to the transcript
- `q` in normal mode closes the current chat shell
- `gr` reopen the active thread inbox
- `gs` open the active thread settings sheet
- `gR` switch between the side rail and the centered overlay
- `g?` or `<F1>` in normal mode opens the shortcut sheet for the current surface
- `<CR>` inserts a newline

Workbench tray defaults:

- `<CR>` inspect the selected fragment
- `dd` remove the selected fragment
- `D` clear the active thread workbench
- `o` open compose review
- `i` open compose review and insert the selected fragment handle
- `p` park the selected active fragment
- `u` unpark the selected parked fragment
- `P` open packet preview
- `g?` or `<F1>` open the shortcut sheet for the current surface
- `q` close the tray

Packet preview shows:

- referenced active fragments that will be sent
- unreferenced active fragments that still need attention
- parked fragments that will remain staged
- the final compiled packet text

Compose review borders also show the live packet state:

- referenced active fragments already placed in the packet
- pending active fragments that still need placement or parking
- parked fragments that remain outside the current packet

Compose review defaults:

- `<C-s>` send the current packet from normal or insert mode
- `gS` send the current packet from normal mode
- `P` preview the compiled packet
- `<Tab>` focus the staged fragment list
- `i` insert the selected fragment handle into the packet template
- `p` park the selected active fragment from the fragment list
- `u` unpark the selected parked fragment from the fragment list
- `g?` or `<F1>` open the shortcut sheet for the current surface
- `q` close compose review

These mappings are configurable through `setup()` and can be disabled by setting
individual mappings to `false`. By default, hidden request dialogs can always be
reopened with `<F2>`.

If `<C-s>` is captured by terminal flow control, remap
`keymaps.composer.send` or run `stty -ixon`.

Secondary viewers like `:CodexInspect`, `:CodexEvents`, request prompts, and thread or smoke reports now share a popup stack. `q` or `<Esc>` closes the latest viewer and returns to the previous one. The side rail and the centered overlay are mutually exclusive alternate views over the same chat state.

Blocking request flows are protocol-first and do not render inline as transcript content. When Codex asks for command approval, file-change approval, or tool input, the plugin opens a dedicated request viewer above the chat overlay in normal mode. Use `:CodexRequest` to reopen the active request if you hide it before responding. File-change approvals also expose a dedicated review surface through `:CodexReview` or the request-local `o` mapping. Within that review, `]f` and `[f` move between changed files and `o` opens a dedicated per-file diff viewer.

To expose Codex state in your own statusline, add:
>
    set statusline+=%{%v:lua.require('neovim_codex').statusline()%}
<

The component reports whether Codex is running, waiting for a request response,
idle, stopped, or in error, and includes a pending-request reopen hint.

Default request viewer mappings:

- the viewer opens in normal mode and stays read-only
- `<CR>` resolve the current request
- `a` approve once when available
- `s` approve for session when available
- `d` decline
- `c` cancel
- `o` open the file-change review when the pending request is a file-change approval
- `g?` or `<F1>` open the shortcut sheet for the current surface
- `q` or `<Esc>` hide the viewer without resolving the request

Default file-change review mappings:

- `o` open the currently selected changed file in a dedicated diff viewer
- `]f` move to the next changed file
- `[f` move to the previous changed file
- `a` approve once
- `s` approve for session
- `d` decline
- `c` cancel
- `g?` or `<F1>` open the shortcut sheet for the current surface
- `q` or `<Esc>` close the top review viewer and return to the previous review layer

MARKDOWN CONTRACT                                     *neovim-codex-markdown*

The transcript, composer, and other plugin-owned markdown surfaces are normal
markdown scratch buffers with these buffer variables:

- `b:neovim_codex = true`
- `b:neovim_codex_role` commonly includes `"transcript"`, `"composer"`,
  `"details"`, `"events"`, `"workbench"`, `"compose_review_message"`, or
  `"compose_review_fragments"`; stacked request and review viewers use their
  own surface-specific roles
- `b:neovim_codex_thread_id = <thread-id>`

This lets normal markdown filetype, treesitter, and render plugins apply
without a custom filetype.

Heading highlight groups can also be overridden in user config:

- `NeovimCodexChatTurnHeading`
- `NeovimCodexChatUserHeading`
- `NeovimCodexChatAssistantHeading`
- `NeovimCodexChatPlanHeading`
- `NeovimCodexChatReasoningHeading`
- `NeovimCodexChatActivityHeading`
- `NeovimCodexChatCommandHeading`
- `NeovimCodexChatFileChangeHeading`
- `NeovimCodexChatToolHeading`
- `NeovimCodexChatReviewHeading`
- `NeovimCodexChatNoticeHeading`

HEALTH                                                 *neovim-codex-health*

Run >
    :checkhealth neovim_codex
<
to verify:

- NeoVim `0.11+`
- the configured Codex executable is available on `PATH`
- `nui.nvim` availability
- app-server start and initialize handshake viability

For a detailed report buffer with recent protocol events, use >
    :CodexSmoke
<

CONFIGURATION                                         *neovim-codex-configuration*

Call >
    lua require('neovim_codex').setup({})
<
from your plugin manager configuration to override defaults.

Global mappings are disabled by default. Buffer-local overlay mappings have
conservative defaults. Fast open/reopen actions use
`keymaps.global_fast_modes`; workflow actions use
`keymaps.global_workflow_modes`. If you only set the older
`keymaps.global_modes`, it still acts as the fallback for both lanes.

WORKBENCH                                               *neovim-codex-workbench*

The workbench is a thread-local staging area for semantic fragments.

Current capture commands:

- `:CodexCapturePath` adds the current file as a `path_ref`
- `:CodexCaptureSelection` adds the current visual selection as a
  `code_range`
- `:CodexCaptureDiagnostic` adds the current diagnostic under cursor as a
  `diagnostic`

Workbench tray defaults:

- `<CR>` inspect the selected fragment
- `dd` remove the selected fragment
- `D` clear the active thread workbench
- `o` open compose review
- `i` open compose review and insert the selected fragment handle
- `p` park the selected active fragment
- `u` unpark the selected parked fragment
- `P` open packet preview
- `g?` or `<F1>` open the shortcut sheet for the current surface
- `q` close the tray

Compose review edits a packet template, not a plain appendix. Staged fragments
receive short handles such as `[[f1]]`. Only referenced handles are expanded
and sent; unreferenced staged fragments block send until they are removed or
referenced.

Compose review defaults:

- `<C-s>` send the current packet from normal or insert mode
- `gS` send the current packet from normal mode
- `P` preview the compiled packet
- `<Tab>` focus the staged fragment list
- `i` insert the selected fragment handle into the packet template
- `p` park the selected active fragment from the fragment list
- `u` unpark the selected parked fragment from the fragment list
- `g?` or `<F1>` open the shortcut sheet for the current surface
- `q` close compose review

Lua API:

- `require("neovim_codex").capture_text_fragment({ label = "Latest test run", text = "...", filetype = "markdown", source = "neotest", category = "runtime" })`
  stages a first-class `text_note` fragment for runtime context, tool output,
  logs, or notes
- workbench capture is code-world first in this slice; chat text can still be copied manually when needed

==============================================================================
LIMITATIONS                                            *neovim-codex-limits*

Current known limits:

- a brand-new empty thread may not appear in `thread/list` yet
- a brand-new empty thread may not be resumable yet
- `thread/read includeTurns=true` can fail before the first user message is
  persisted; the plugin falls back to a metadata-only read for reports
- raw protocol stays in `:CodexEvents`; the main transcript stays terse and
  `:CodexInspect` exposes verbose detail on demand

==============================================================================
DOGFOOD LOOP                                            *neovim-codex-dogfood*

When iterating locally with `lazy.nvim`, the expected loop is: >
    :Lazy reload neovim-codex
    :checkhealth neovim_codex
    :CodexChat
    :CodexThreadNew
<
Then write a short prompt in the composer and send it with `<C-s>`.
